# VSTOについて

Word の採点チェッカーを正確かつ厳密にするために VSTO を利用する際の説明と、運用方針のまとめです。

---

## 1. VSTOとは何か、なぜ正確かつ厳密に採点できるのか

**VSTO（Visual Studio Tools for Office）** は、.NET（C# など）で **Word / Excel / PowerPoint のアドイン**を作る技術です。  
Word 起動時にアドインが読み込まれ、リボンやコマンドの「実行」をフックできます。

**なぜ採点が正確・厳密になるか**

- **現状（結果だけ見る採点）**  
  ドキュメントの**最終状態**（テキスト・スタイル・書式など）を Word OM で読んで判定する。  
  → 「正解の手順でやったか」は分からない。

- **VSTO を使うと**  
  アドインが **「どのコマンドがいつ実行されたか」をログに記録**できる。  
  → 「編集記号を非表示→再度表示」なら **ShowAll が 2 回**実行されたか、「切り取り」なら **Cut** が実行されたか、といった**手順・過程**を判定できる。

例（`ID,問題概要,VSTO推奨理由,監視対象コマンド例.md` より）:
- **1-1**: 結果は「表示 ON」だが、**2 回押したか**はファイル解析では分からない → VSTO で `ShowAll` の実行回数を記録。
- **2-1**: 見た目は「移動」だが、**Cut を使ったか / コピー＋削除か**は結果だけでは区別できない → VSTO で `Cut` / `Paste` を記録。

「**過程必須**」な問題は、VSTO でコマンド実行をログに残すことで、正確かつ厳密に採点できる。

---

## 2. 科目ごとに VSTO を作成する必要があるか

**はい。科目（アプリ）ごとに別アドインが必要です。**

- **Word** → Word 用 VSTO アドイン（例: MOSWordVSTOAddIn）
- **Excel** → Excel 用 VSTO アドイン
- **PowerPoint** → PowerPoint 用 VSTO アドイン

理由:
- 各アプリのアドインは、そのアプリのプロセスに読み込まれる。
- リボンやコマンドの ID（idMso）は **アプリごとに別**。
- 1 つの DLL を「Word と Excel の両方で動かす」ような共通アドインは作れない。

Word の採点を厳密にするなら Word 用 VSTO を 1 つ作り、Excel / PowerPoint も同様に厳密にするなら、それぞれ用の VSTO を別プロジェクトで用意する。

---

## 3. VSTO を作成する際のフォルダ・配置場所

**Word 用**は、`MOS Word app` の直下に VSTO 専用フォルダを置く構成がよい（依頼書・既存プロジェクトと一致）。

```
MOS Word app/
└── MOSWordVSTOAddIn/                 # VSTOアドイン専用プロジェクト
    ├── MOSWordVSTOAddIn.sln
    ├── MOSWordVSTOAddIn.csproj
    ├── Logger.cs, Ribbon.cs, Ribbon.xml, ThisAddIn.cs など
    └── bin/Debug/                    # .dll, .manifest, .vsto
```

Excel / PowerPoint 用も、それぞれのアプリ用ソリューション直下に専用フォルダ（例: `MOSExcelVSTOAddIn` / `MOSPowerPointVSTOAddIn`）を用意する形が分かりやすい。

## 4. VSTO 開発のベイビーステップ

1. **環境**  
   Visual Studio で「Office/SharePoint 開発」ワークロードを有効にし、テンプレート「Word 2013 および 2016 VSTO アドイン」でプロジェクト作成（既存の MOSWordVSTOAddIn を流用しても可）。

2. **最小動作**  
   ビルドし、Word 起動時にアドインが読み込まれるか確認。`ThisAddIn_Startup` に `MessageBox.Show("読み込みました");` を入れて F5 で確認。

3. **1 コマンドだけフック**  
   まず **1-1 の ShowAll** だけを Ribbon XML の `<commands>` でフックし、`onAction` でコマンド ID を受け取り、Logger で `%TEMP%/mos_word_log.txt` に 1 行書き出す。Word で「編集記号の表示/非表示」をクリックし、ログに `ShowAll` が記録されるか確認。

4. **採点との連携**  
   既存の `LogReader` がそのログを読んで「ShowAll が 2 回実行されたか」を判定できるようにする（ログ形式は依頼書の `[yyyy-MM-dd HH:mm:ss] [CommandID] Executed` に合わせる）。

---

## 5. 過程ログが必要な問題と結果だけでよい問題の使い分け

| 問題の種類 | 判定に必要なもの | 使うもの | 例 |
|------------|------------------|----------|-----|
| **過程が本質** | 「どのコマンドを・何回実行したか」 | **VSTO**（コマンド実行をログ） | 1-1（ShowAll 2回）、2-1（Cut）、8-1（目次2） |
| **結果だけでよい** | 「今の文書の状態が正しいか」 | **チェッカー**（Word OM でドキュメントを読む） | 1-5（書式クリア）、1-3（参照2）、1-4（見出し2） |

- **1-5（書式のクリア）**  
  「会社の社会的責任」の段落が標準スタイルで太字・斜体・下線なしか、という**結果**を見れば十分 → **WordChecker のまま**でよい。VSTO は不要。

- **1-1（編集記号を非表示→再度表示）**  
  最終状態は「表示 ON」なので、**結果だけでは「2回押したか」が分からない** → **VSTO で ShowAll の実行をログ**し、採点時にそのログを参照する。

**採点フロー**  
- 過程を見る問題 → VSTO のログ（例: `LogReader.HasCommandExecutedAtLeast("ShowAll", 2)`）で判定。  
- 結果だけ見る問題 → 従来どおり **WordChecker** がドキュメントを開いてスタイル・書式・テキストをチェック。

1 問ごとに「VSTO を使うか / チェッカーだけか」を決める。両方を使い分ける。

---

## 6. 今後も使えるようにするには

- **VSTO**  
  「過程が必要な問題用のコマンドログ」に特化。新しい問題で**過程が本質**なら、そのコマンドを VSTO の監視対象に追加し、採点側でそのログを読む処理を追加する。

- **チェッカー（WordChecker）**  
  「結果だけ見ればよい問題」はすべてこちらで担当。新しい「結果だけ」問題が増えても、既存の 1-5 と同様、チェッカーを拡張していく。

- **新しい問題を追加するとき**  
  - **過程が本質** → VSTO にそのコマンドを 1 つ追加し、採点でログを参照。  
  - **結果だけでよい** → VSTO には触れず、WordChecker に新しいチェック（新しいタスク用のメソッド）を追加する。

この方針にしておけば、結果のみでいい問題はチェッカーだけでよく、過程をログする問題だけ VSTO を活用でき、新しい問題を足すときも同じ設計で拡張できる。
